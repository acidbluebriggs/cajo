The <Unnamed> package is used startup and configure the server.

The cajo SDK is used for transparent distributed computing between remote Java Virtual Machines, providing client user interfaces via Applets and WebStart. Through the cajo grail mechanism, {@link cajo.sdk.AbstractService services} can dynamically discover and use each other, by specified functionality. Also, via the cajo CodebaseServer mechanism; rich distributed user interfaces to the service objects can be provided, services can also provide {@link cajo.sdk.AbstractController controller} objectst to themselves, and services can send {@link cajo.sdk.AbstractAgent agent} objects to other services for themselves. Full cajo framework Javadoc is available, as is an introductory overview.

The intended use of this SDK is as a 'starting point' for new projects. Feel free to overwrite it, and add classes to it, and when you are ready to create another server, simply start with an original copy of the SDK again. Later, if you wish to take advantage of some of the more advanced features of the full cajo library, simply replace grail.jar with cajo.jar. The grail.jar library is a proper subset of cajo.jar.

The cajo grail framework allows Java Virtual Machines to dynamically distribute their functionality, and user interfaces, across the network transparently. It essentially creates a seamless continuum between cooperating remote Java Virtual Machines, or in other words, a World Wide Virtual Machine. (wwvm) The classes defined in this SDK, along with the included Cajo object, can provide full functional completeness. A single JVM can furnish more than one service/controller/view combination, if it wishes.

An overview of the fundamental architecture:

A typical cajo server deployment consists minimally of the following nine files:

  • grail.jar - the standard cajo grail library
  • server.jar - the implementation(s) of the service(s), and optional main application
  • controller.jar - the hot swappable implementation(s) of the controler(s), and optional view(s)
  • agent.jar - the hot swappable implementation(s) of the agent(s)
  • client.jar - the standard cajo controller view GUI viewer
  • favicon.ico - to give the Applet a little style
  • splash.jpeg - to give the WebStart app a little style (gif or png too)
  • icon.gif - to provide the WebStart app its own icon (png or jpeg too)
  • server.policy - a security policy for the server JVM

  • optionally, any other library jars needed by either controllers, or services.*

A hot-swappable controller and agent jar means that new user interfaces, controllers, and agents can be developed for currently running services. Simply overwriting the jar file will cause all subsequent clients to use the new controllers/views/agents.

*The optional CodebaseServer utility class can be used to send any mobile code .jar files, for agents and controllers, from its immediate subdirectories, with the notable exception of server.jar. Jars used only by the server are conventionally renamed with the .zip extension, to stop CodebaseServer from sending them. The .zip and .jar file formats are identical as far as the Java runtime environment is concerned.

To view the service client in a full-browser Java Applet, visit the http service, e.g. http://myhost:port/name (name being replaced by the actual name of the service to be used)

The application can also be started via WebStart by visiting the server thusly: http://myhost:port/name! (note the addition of the exclamation mark!) Following the WebStart launch, the system will ask if you wish a shortcut icon placed on the desktop. If yes, clicking on this icon can be used for all subsequent client application launches for that service.

For highest performance, the client can be run as a standard Java application, via the following invocation:

java -cp client.jar:grail.jar -Djava.security.manager -Djava.security.policy=client.policy cajo.sdk.Client //myhost:port/name
(replace : with ; for windows users)

Notes from a network perspective:

  • Servers need to have two open inbound ports (typically 80 and 1196) The first is optional, and only needed if the service is furnishing controllers, agents, or other mobile code classes.
  • Clients can operate with all of their inbound ports closed
  • Servers and clients both can be behind NAT routers

The grail.jar library also contains three commonly used cajo classes, not demonstrated in the SDK example:

  • ItemProxy and ClientProxy, these are used to allow callback method invocations to JVMs which are firewalled.
  • Xfile is used to pass files between JVMs.
